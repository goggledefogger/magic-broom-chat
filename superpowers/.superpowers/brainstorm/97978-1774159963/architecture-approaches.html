Architecture Approaches

Three ways to build Magic Brooms — click to select your preference

A

React + Supabase (Recommended)

Frontend: React (Vite) + TailwindCSS

Backend: Supabase (Postgres + Auth + Realtime)

Real-time: Supabase Realtime (WebSocket, built-in)

Pros

  • Least backend code — auth, database, and real-time all provided
  • Free tier covers a small group easily
  • Postgres row-level security for permissions
  • Full-text search built into Postgres
  • Fast to MVP — focus on UI, not infrastructure

Cons

  • Vendor dependency on Supabase
  • Less control over backend logic
  • Real-time has some latency vs raw WebSockets
B

React + Express + Socket.io + SQLite

Frontend: React (Vite) + TailwindCSS

Backend: Express.js REST API

Real-time: Socket.io (WebSocket)

Database: SQLite (via better-sqlite3)

Pros

  • Full control over everything
  • No external dependencies — runs entirely local
  • SQLite is zero-config, file-based
  • Socket.io gives instant real-time
  • Most educational — see all the layers

Cons

  • More code to write (auth, API, WebSocket handling)
  • SQLite limits concurrency (fine for small group)
  • Must build auth from scratch or add passport.js
C

Next.js + Prisma + Postgres + Pusher

Frontend + Backend: Next.js (App Router, Server Actions)

ORM: Prisma

Database: Postgres (local Docker)

Real-time: Pusher or Ably (hosted WebSocket service)

Pros

  • Modern full-stack in one framework
  • Prisma gives type-safe database access
  • Server Actions reduce API boilerplate
  • Good deployment story (Vercel)

Cons

  • Requires Docker for local Postgres
  • Pusher/Ably adds another vendor dependency
  • Next.js App Router has a learning curve
  • Most complex setup of the three
